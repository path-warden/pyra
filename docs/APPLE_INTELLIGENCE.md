# Apple Intelligence Provider

pyra can summarize documents using Apple's on-device Foundation Models
framework (macOS 26 Tahoe). When enabled, the engine routes `--summarize=llm`
calls directly through the Foundation Models CGo bridge, with no network, no
HTTP server, and no API key.

The provider is **opt-in** via the `applefm` build tag. Default builds keep
a stub that reports unavailable, so non-Mac, Intel-Mac, CGo-disabled, and
non-Tahoe builds work unchanged and fall back to Ollama or extractive
summarization.

## Requirements

- macOS 26 Tahoe or later
- Apple Silicon (M1 / M2 / M3 / M4 series)
- Apple Intelligence enabled (Settings → Apple Intelligence & Siri)
- Xcode 15+ with the macOS 26 SDK
- `CGO_ENABLED=1`

## How it works

The provider lives in `internal/summarize/llm/apple_darwin_fm.go` and is
built only when **all** of these constraints hold:

```
//go:build darwin && arm64 && cgo && applefm
```

It wraps [`github.com/blacktop/go-foundationmodels`](https://github.com/blacktop/go-foundationmodels),
which compiles a Swift shim (`FoundationModelsShim.swift`) into a static
library (`libFMShim.a`) that links into the pyra binary. The shim is
**not** vendored or shipped pre-built; it has to be compiled locally
because:

1. `libFMShim.a` is gitignored upstream, so the Go module cache never
   contains it.
2. The module cache is read-only; we cannot `go generate` in place.

The workflow below works around both by vendoring the dependency into the
repo's `vendor/` tree, where `go generate` is allowed to write.

## Build workflow

From the repo root:

```bash
# 1. Vendor dependencies so the Swift shim can be compiled in place.
go mod vendor

# 2. Compile the Swift bridge into libFMShim.a inside vendor/.
cd vendor/github.com/blacktop/go-foundationmodels
go generate ./...
cd -

# 3. Build pyra with the applefm tag against the vendored tree.
CGO_ENABLED=1 go build -mod=vendor -tags=applefm -o pyra ./cmd/pyra
```

The first two steps only need to be repeated when the
`go-foundationmodels` version in `go.mod` changes.

## Verifying

After install, run:

```bash
./pyra import --summarize=llm <source>
```

The changelog (`<bundle>/changelog.md`) will record `summarize_mode: llm`,
and the bundle's summary callouts will be generated by Foundation Models.
If Apple Intelligence is disabled or the device is ineligible at runtime,
the engine silently falls back to Ollama (if configured) and finally to
extractive summarization.

## Why not just use the HTTP server alternative?

An [Apple Intelligence HTTP wrapper](https://github.com/gouwsxander/Apple-Intelligence-API)
exists that exposes Foundation Models as an OpenAI-compatible server. It
works with pyra's existing `api` provider (set `api_endpoint:
http://localhost:8080/api/v1/chat/completions` in `llm.config`), but
requires running a separate Swift server alongside the CLI.

The CGo-bridged provider documented here removes that operational step:
one binary, in-process, no port to manage. The tradeoff is the extra
build-time setup above.

## Disabling

To go back to the stub, just build without the tag:

```bash
go build -o pyra ./cmd/pyra
```

The provider falls through and the engine picks Ollama or extractive.
